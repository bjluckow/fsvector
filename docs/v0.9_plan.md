# v0.9 — Pipeline Rewrite: Batch Processing & Concurrent Workers

## Goal

Replace the sequential file-at-a-time pipeline with a concurrent,
queue-based architecture where independent workers (embed, caption,
OCR, transcribe, text-embed) run in parallel, each pulling from its
own channel and flushing work in batches. Files are downloaded and
extracted once, then routed to only the workers that still have
outstanding work, determined by inspecting existing DB artifacts.

**Target:** 8,600 images from ~7 hours → ~45 minutes (10x+).

---

## Architecture Summary

The pipeline has two distinct phases:

**Phase 1 — Index:** Download each file, convert to model-friendly
formats, create `files` and `items` rows in the DB, assess which
processing stages are still needed, and enqueue work items into
the appropriate worker queues. After phase 1, the database knows
about every file and every item that exists. This phase is
modality-aware — each extractor knows how to break its file type
into items.

**Phase 2 — Process:** Workers are already running as goroutines.
They are modality-agnostic — they just see batches of things that
need a specific operation (embed, caption, OCR, etc.). They call
services, get results, and write `chunks` rows to the DB. When a
worker produces derived data (OCR text, transcript text), it
enqueues new work items into downstream queues (text_embed).

```
 ┌──────────────────────────────────────────────────────────────┐
 │               Phase 1 — Index (extraction)                  │
 │  per-modality: download → convert → UpsertFile → UpsertItem │
 │  → assess needed stages → enqueue WorkItems                 │
 │  bounded concurrency via sourcegraph/conc pool              │
 └────────┬──────────┬──────────┬──────────┬──────────┬────────┘
          │          │          │          │          │
          ▼          ▼          ▼          ▼          ▼
 ┌─────────────────────────────────────────────────────────────┐
 │               Phase 2 — Process (workers)                   │
 │                                                             │
 │  ┌─────────┐ ┌───────┐ ┌───────┐ ┌─────────┐ ┌─────────┐  │
 │  │  embed  │ │caption│ │  ocr  │ │transcr. │ │txt embed│  │
 │  │  queue  │ │ queue │ │ queue │ │  queue  │ │  queue  │  │
 │  └────┬────┘ └───┬───┘ └───┬───┘ └────┬────┘ └────┬────┘  │
 │       ▼          ▼         ▼          ▼           ▼        │
 │  ┌─────────┐ ┌───────┐ ┌───────┐ ┌─────────┐ ┌─────────┐  │
 │  │  batch  │ │ batch │ │parall.│ │ parall. │ │  batch  │  │
 │  │ worker  │ │worker │ │worker │ │ worker  │ │ worker  │  │
 │  └────┬────┘ └───┬───┘ └───┬───┘ └────┬────┘ └────┬────┘  │
 │       │          │     ocr text    transcript      │       │
 │       │          │     ────┬────   ────┬─────      │       │
 │       │          │         └───────────┘            │       │
 │       │          │              │                   │       │
 │       │          │       ┌──────▼──────┐            │       │
 │       │          │       │  txt embed  │◄───────────┘       │
 │       │          │       │   queue     │                    │
 │       │          │       └──────┬──────┘                    │
 │       ▼          ▼              ▼                           │
 │  ┌──────────────────────────────────────┐                   │
 │  │         upsert worker               │                   │
 │  │  batched UpsertChunkBatch calls      │                   │
 │  └──────────────────────────────────────┘                   │
 └─────────────────────────────────────────────────────────────┘
```

**DB writes by phase:**
- Phase 1 (extractors): `UpsertFile` → `UpsertItem` — creates the
  file and item rows so workers have valid IDs to write against.
- Phase 2 (workers): `UpsertChunkBatch` — each worker writes chunk
  rows (embeddings, captions, OCR text, etc.) against the item IDs
  established in phase 1.

**Key principles:**
- Workers are decoupled. Each runs its own goroutine, accumulates
  items until batch-full or flush-timeout, then makes one service call.
- Workers never deal with skip logic. If they receive a work item,
  it needs processing. All skip decisions happen in phase 1.
- Shared memory via `FileData` with atomic refcounting. Data is freed
  when the last worker releases its reference.
- Skip logic uses Option A: infer completed stages from existing
  `items`/`chunks` rows. No new status columns.
- Modality and Stage are typed constants. `ModalityStages` maps each
  modality to its ordered stage list. Workers and routing are
  data-driven, not switch statements.
- Conversion (HEIC→JPEG, DOCX→text, video→frames+audio) happens in
  phase 1 via the existing convertsvc. No batch endpoints needed for
  conversion — concurrency comes from multiple extractors calling it
  simultaneously.
- `sourcegraph/conc` for bounded concurrency in extraction. Raw
  channels + a custom `BatchWorker` for the accumulate-and-flush
  pattern.

## Package Layout

```
internal/
├── pipeline/           # orchestration: FSM, workers, routing, queues
│   ├── stage.go        # Stage, Modality consts, ModalityStages, StageDeps
│   ├── item.go         # WorkItem, FileData (refcounted shared data)
│   ├── worker.go       # BatchWorker: accumulate → flush → route downstream
│   ├── router.go       # StageQueues map, Route()
│   ├── pipeline.go     # Pipeline struct, RunBatch, worker wiring
│   ├── assess.go       # skip logic: which stages are already done
│   └── mapping.go      # extension → Modality (carried over from current)
│
├── process/            # actual work: one file per processing operation
│   ├── extract.go      # per-modality extractors (Image, Text, Audio, Video, Email)
│   ├── embed.go        # BatchClipEmbed, BatchTextEmbed
│   ├── caption.go      # BatchCaption
│   ├── ocr.go          # OCR (single-item, called concurrently)
│   ├── transcribe.go   # Transcribe (single-item, called concurrently)
│   └── convert.go      # ConvertToText, ConvertToImage wrappers
│
├── clients/            # HTTP clients to Python services (exists, modified)
│   ├── http.go         # shared doJSON, buildMultipart helpers
│   ├── embed.go        # + EmbedImageBatch
│   ├── vision.go       # + CaptionBatch (OCR unchanged)
│   ├── convert.go      # unchanged
│   └── transcribe.go   # unchanged
│
└── store/              # DB layer (exists, modified)
    ├── write.go        # UpsertFile, UpsertItem, UpsertChunkBatch
    ├── assess.go       # GetFileStatus (infer completed stages)
    └── ...             # existing read/search code unchanged
```

### M0.9.3 — Process package: real service calls behind clean functions

**Goal:** The `process/` package wraps each service call with the
correct WorkItem input/output contract. Each function is
independently testable against running services.

**process/embed.go:**

- [ ] `BatchClipEmbed(ctx, client, model, batch []*WorkItem) error`
  - reads `item.FileData.Data`, calls `client.EmbedImageBatch`
  - writes results to `item.Embedding`
  - handles nil (failed) embeddings per-item gracefully

- [ ] `BatchTextEmbed(ctx, client, model, batch []*WorkItem) error`
  - reads `item.Text`, calls `client.EmbedTexts`
  - writes results to `item.Embedding`

**process/caption.go:**

- [ ] `BatchCaption(ctx, client, batch []*WorkItem) error`
  - reads `item.FileData.Data`, calls `client.CaptionBatch`
  - writes results to `item.Text`

**process/ocr.go:**

- [ ] `OCR(ctx, client, item *WorkItem) error`
  - single-item call to `client.OCR`
  - writes result to `item.Text`
  - called concurrently by the OCR worker (not batched)

**process/transcribe.go:**

- [ ] `Transcribe(ctx, client, item *WorkItem) error`
  - single-item call to `client.Transcribe`
  - writes result to `item.Text`

**process/extract.go:**

- [ ] `Extractor` interface:
      `Extract(ctx, fi, status) (*FileData, []*WorkItem, error)`
      Each extractor is responsible for:
      1. Download + convert (phase 1 — modality-specific)
      2. `store.UpsertFile` → get fileID, store on FileData
      3. `store.UpsertItem` for each item → get itemID, store on WorkItem
      4. Assess which stages are already done (via status)
      5. Return only WorkItems for stages that still need processing
      Workers receive items with valid fileID/itemID already set,
      so they only need to call `UpsertChunkBatch` with results.
- [ ] `ImageExtractor`: download, convert if needed, UpsertFile,
      UpsertItem (type="image"), return work items for clip_embed +
      caption + ocr (minus already-done stages)
- [ ] `TextExtractor`: download, convert to text, chunk, UpsertFile,
      UpsertItem (type="text"), return work items for text_embed
- [ ] `AudioExtractor`: download, UpsertFile, UpsertItem
      (type="audio"), return work item for transcribe
- [ ] `VideoExtractor`: download, extract frames + audio track via
      convertsvc (/convert/video/frames, /convert/video/audio),
      UpsertFile, UpsertItem per frame (type="frame") + UpsertItem
      for audio (type="audio_track"), return work items for
      clip_embed + caption per frame, transcribe for audio
- [ ] `EmailExtractor`: download, parse via convertsvc
      (/convert/email), UpsertFile, UpsertItem for body
      (type="body"), delegate attachments to sub-extractors which
      create their own items

**process/convert.go:**

- [ ] Thin wrappers: `ConvertToText`, `ConvertToImage` calling
      existing convert client

**Deliverable:** Integration tests against running Docker services.
Each process function gets a test that sends real data to real
services and verifies output fields are populated.

### M0.9.4 — Wire it up: Pipeline.RunBatch for images

**Goal:** End-to-end batch indexing works for image files. This is
the highest-impact modality and validates the full architecture
before we extend to other modalities.

**pipeline/pipeline.go:**

- [ ] `Pipeline` struct holds: `StrategyConfig`, `StageQueues`,
      extractors, store, clients
- [ ] `NewPipeline(cfg) *Pipeline` wires up all channels and workers
- [ ] `RunBatch(ctx, files []source.FileInfo) error`:
  Phase 1 — Index:
  1. Start all BatchWorkers as goroutines (via errgroup)
  2. Group files by modality
  3. For each file (bounded concurrency via `conc.Pool`):
     a. Extract: download, convert, UpsertFile, UpsertItem(s)
     b. Assess: query existing chunks, determine needed stages
     c. Enqueue: push WorkItems (with fileID/itemID set) into
        stage queues for only the stages that need work
  4. Close input queues when extraction is done
  Phase 2 — Process (already running):
  5. Workers flush remaining items, cascade-close downstream
     queues (OCR/caption → text_embed → upsert)
  6. Wait for all workers to finish

- [ ] Worker wiring for image flow:
  - `embedWorker`: batchSize=32, calls `process.BatchClipEmbed`,
    routes results to upsert
  - `captionWorker`: batchSize=16, calls `process.BatchCaption`,
    spawns downstream text_embed items for non-empty captions,
    routes to upsert
  - `ocrWorker`: runs N concurrent OCR calls (not batch), spawns
    downstream text_embed items for OCR text, routes chunks to
    text_embed
  - `textEmbedWorker`: batchSize=64, calls `process.BatchTextEmbed`,
    routes to upsert
  - `upsertWorker`: batchSize=50, calls `store.UpsertChunkBatch`
    (item rows already exist from phase 1 extraction)

- [ ] Downstream queue lifecycle: text_embed and upsert queues stay
      open until all upstream workers have exited, then close

- [ ] `Process(ctx, fi) (Result, error)` — single-file path for
      watch events. Extracts + enqueues into a temporary single-use
      pipeline, waits for completion. (Can be a thin wrapper that
      calls RunBatch with a one-element slice.)

**Testing:**

- [ ] End-to-end test: 10 test images (mix of jpg, png, heic) →
      RunBatch → verify DB state after both phases:
  - Phase 1: file row exists with correct hash, image item row
    exists (created by extractor)
  - Phase 2: embed chunk exists (non-null embedding), caption
    chunk exists (non-null text_content), ocr item exists
    (text may be empty for photos, that's ok)

- [ ] Skip test: run same 10 images again → verify no service calls
      made (all stages already complete)

- [ ] Partial skip test: delete caption chunks for 3 files, re-run →
      verify only captioning runs for those 3, embed/OCR skipped

- [ ] Benchmark: 100 images, compare wall time vs old sequential
      pipeline. Target: 3x+ improvement minimum.

**Deliverable:** `reindex` command works for image-only datasets
using the new pipeline. Old sequential path remains available via
config flag as fallback.

## What This Does Not Include

- GPU/MPS acceleration — separate project, but the batch endpoints
  are designed to benefit from it with no API changes
- Streaming progress during batch — future, would use SSE or
  websocket from the API server
- Priority queuing (watch events jump ahead of reindex) — future,
  would add priority field to WorkItem and a priority channel
- Distributed processing across machines — future, would replace
  channels with a message broker
- Schema migrations — the existing schema supports this design
  as-is with no changes needed
