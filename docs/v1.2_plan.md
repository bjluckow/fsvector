# fsvector v1.2c — Text Chunking

## Overview

All text files are currently embedded as a single vector at `chunk_index=0`.
Content beyond the first 4096 characters is silently truncated. For long
documents this means only the beginning of the file is searchable.

Chunking splits extracted text into overlapping windows, embeds each
separately, and stores one row per chunk. Search returns the best-scoring
chunk per file so the user sees one result per file regardless of chunk count.

---

## Config

Three new vars in `internal/config/config.go` and `.env.example`:
```
CHUNK_SIZE=1000       # characters per chunk
CHUNK_OVERLAP=100     # overlap between consecutive chunks
MIN_CHUNK_SIZE=100    # skip chunks shorter than this
```

Character-based rather than token-based. Reliable approximation:

| Model                          | Token limit | Approx chars |
|-------------------------------|-------------|--------------|
| all-MiniLM-L6-v2              | 256         | ~1000        |
| all-mpnet-base-v2             | 384         | ~1500        |
| CLIP text encoder             | 77          | ~300         |

Default targets `all-MiniLM-L6-v2`.

## New Package: `internal/chunk/`

Pure text splitting, no external dependencies.
```go
package chunk

// Split splits text into overlapping chunks of at most size characters,
// with overlap characters of context carried over between chunks.
// Chunks shorter than minSize are discarded.
func Split(text string, size, overlap, minSize int) []string
```

### Algorithm
```
1. Normalize whitespace
2. i = 0
3. while i < len(runes):
     end = min(i + size, len(runes))
     chunk = runes[i:end]
     snap end to nearest whitespace within 50 chars to avoid mid-word cuts
     if len(chunk) >= minSize:
         append chunk
     i += size - overlap
4. return chunks
```

Whitespace snapping avoids embedding half-words at chunk boundaries.

## Pipeline Changes

### `internal/pipeline/pipeline.go`

`processText` replaces the single embed call with a chunked loop.

`pipeline.Config` gains:
```go
ChunkSize    int
ChunkOverlap int
MinChunkSize int
```

`Result` changes to support multiple chunks:
```go
type Result struct {
    Files      []store.File
    Skipped    bool
    SkipReason string
}
```

Callers in `reconcile()` and `handleEvents()` loop over `result.Files`
and upsert each.

### Chunk embed loop
```
extracted text
  → chunk.Split(text, cfg.ChunkSize, cfg.ChunkOverlap, cfg.MinChunkSize)
  → for each chunk i:
      → embed chunk
      → build store.File with chunk_index=i
      → collect into result.Files
```

## Store Changes

### `store/upsert.go`

No changes — `chunk_index` is already a parameter and the unique index
is already `(path, chunk_index, embed_model)`.

### `store/delete.go` — new: `DeleteStaleChunks`

When a file is re-indexed with fewer chunks than before, old chunk rows
need cleaning up:
```go
// DeleteStaleChunks hard-deletes chunks with index >= newChunkCount.
// Called after re-indexing to remove chunks that no longer exist.
func DeleteStaleChunks(ctx context.Context, conn *pgx.Conn,
    path, embedModel string, newChunkCount int) error
```

Note: this is a hard delete, not a soft delete — stale chunks are not
filesystem deletions, they are index artifacts. Soft-delete is reserved
for files removed from the watched directory.

## Search Changes

### `search/search.go`

`DISTINCT ON (path)` returns only the best-scoring chunk per file.
`ORDER BY` must start with the `DISTINCT ON` expression:
```sql
SELECT DISTINCT ON (path)
    path,
    modality,
    file_ext,
    size,
    1 - (embedding <=> $1) AS score,
    indexed_at,
    file_modified_at
FROM files
WHERE deleted_at IS NULL
  AND canonical_path IS NULL
  AND embedding IS NOT NULL
ORDER BY path, embedding <=> $1
LIMIT $2 OFFSET $3
```

`LIMIT` and `OFFSET` apply after deduplication — `--limit 10` always
returns 10 distinct files.

## Store Changes

### `store/delete.go`

`SoftDelete` already deletes all chunks via `WHERE path = $1` — no

## fsvector CLI

No flag or output changes. `fsvector show` chunk count now reflects
the real chunk count rather than always returning 1.

---

## Re-indexing

Existing files indexed before chunking have only `chunk_index=0` with
truncated content. Since content hashes have not changed, the
reconciliation loop will not re-index them automatically. Wipe the
volume and restart `fsvectord` to re-index from scratch with chunking
enabled.
