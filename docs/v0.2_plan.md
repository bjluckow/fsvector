# fsvector v0.2 Plan — Enhanced Search & Filtering

## Goals

Improve the `fsvector search` and `fsvector ls` commands to support
pagination, filtering, and per-modality score normalization. Migrate
read logic out of `store/` into a dedicated `search/` package. Introduce
clean type boundaries between read and write concerns.

---

## Problems Being Solved

### From known_issues.md
- KI-001: Short files produce noisy embeddings — add minimum content length filter
- KI-002: Text-to-image score gap — normalize scores per modality
- KI-003: Corrupt PDFs silently fail — improve error reporting

### New in v0.2
- No pagination — large indexes return top-N with no way to page through results
- No filtering — can't restrict search by modality, extension, date, or size
- No score context — a score of 0.72 means nothing without knowing the range
- Read and write concerns are mixed in `store/` — `store/query.go` handles
  searching and listing but lives alongside upsert and delete logic

---

## Package Restructure

The primary structural change in v0.2 is moving all read logic out of
`store/` into a new `search/` package. Both packages continue to use `pgx`
directly — this is a separation of concerns, not a backend abstraction.

**Before:**
```
internal/
└── store/
    ├── upsert.go
    ├── delete.go
    ├── migrate.go
    └── query.go      ← read logic lives here alongside write logic
```

**After:**
```
internal/
├── store/            ← writes only, used by fsvectord
│   ├── upsert.go
│   ├── delete.go
│   └── migrate.go
├── search/           ← reads only, used by fsvector CLI
│   ├── search.go     ← SearchQuery, SearchResult, Search()
│   ├── list.go       ← List(), Show(), Stats()
│   └── normalize.go  ← score normalization, no pgx
└── parse/
    └── parse.go      ← date/size/duration parsing utilities, no pgx
```

`fsvectord` imports `store/` only.
`fsvector` CLI imports `search/` and `parse/` only.
Neither binary imports both.

## SearchQuery Type

All search parameters are expressed as a single `SearchQuery` struct.
All fields except `Vector` and `Limit` are optional.
```go
type SearchQuery struct {
    Vector  []float32
    Limit   int
    Offset  int

    // filters
    Modality string     // "text" | "image" | ""
    Ext      string     // e.g. "pdf" | ""
    Source   string     // "local" | "s3://..." | ""
    Since    *time.Time // file_modified_at >= Since
    Before   *time.Time // file_modified_at <= Before
    MinSize  *int64     // bytes
    MaxSize  *int64     // bytes
    MinScore *float64   // exclude results below this threshold (raw score)
}
```

## Pagination

All queries support limit/offset pagination. CLI flags:
```
--limit  -n  int   number of results (default: search=10, ls=100)
--offset     int   skip first N results (default 0)
--page   -p  int   1-indexed page shorthand (offset = (page-1)*limit)
```

## Filter Flags

### `fsvector search`
```
<query>              positional, required
--limit    -n  int   number of results (default 10)
--page     -p  int   page number (default 1)
--modality -m  str   filter by modality: text, image
--ext          str   file extension, e.g. pdf
--since        str   modified after: 7d, 30d, 2024-01-01
--before       str   modified before: 2024-01-01
--min-size     str   minimum file size: 10kb, 5mb
--max-size     str   maximum file size: 10mb, 1gb
--min-score    f64   exclude results below this score
--source       str   filter by source: local, s3://...
```

### `fsvector ls`
```
--deleted          include soft-deleted files
--modality -m str  filter by modality
--ext          str  filter by extension
--since        str  modified after
--before       str  modified before
--source       str  filter by source
--limit   -n  int   number of results (default 100)
--page    -p  int   page number (default 1)
```

## Score Normalization

Raw cosine scores are not comparable across modalities. A text result
scoring 0.72 and an image result scoring 0.24 may both be equally
relevant within their modality. v0.2 adds a normalized score computed
within each modality after the query returns.

Normalization is min-max within each modality group:
```
norm = (score - min) / (max - min)
```

Both scores are shown in output:
```
SCORE   NORM   MODALITY  EXT   SIZE     PATH
0.8900  1.000  text      pdf   113KB    resume.pdf
0.7612  0.742  text      pdf   116KB    old_resume.pdf
0.2689  1.000  image     webp  57KB     dog.webp
0.2220  0.732  image     png   74KB     me.png
```

Normalization lives in `search/normalize.go` with no pgx dependency —
it operates purely on `[]SearchResult` after the query returns.

`--min-score` always filters on raw score, not normalized score.

## Date/Size Parsing (`internal/parse/parse.go`)

### Duration/date strings
- `7d` — 7 days ago from now
- `30d` — 30 days ago from now
- `2024-01-01` — absolute date (UTC midnight)
- `2024-01-01T15:04:05` — absolute datetime

### Size strings
- `1024` — raw bytes
- `10kb`, `10KB`, `10kB` — kilobytes
- `5mb`, `5MB` — megabytes
- `1gb`, `1GB` — gigabytes

Both parsers are pure functions with no dependencies beyond stdlib.

## KI-001 Fix: Minimum Content Length

In `internal/pipeline/pipeline.go`, skip files whose extracted text
content is under a configurable minimum before embedding:
```go
if len(strings.TrimSpace(text)) < cfg.MinEmbedLength {
    return Result{
        Skipped:    true,
        SkipReason: fmt.Sprintf("content too short (< %d chars)", cfg.MinEmbedLength),
    }, nil
}
```

Configured via `MIN_EMBED_LENGTH` env var, default 50. Added to
`internal/config/config.go` and `.env.example`.
